docker面试题
docker是什么？
docker是容器技术的典型代表，容器技术同样可以做到应用的隔离，而且容器技术比虚拟化技术更方便，更小巧，因为容器技术不需要虚拟化这一层，也不需要Guest OS这一层（虚拟机操作系统），容器技术直接在本地操作系统之上实现app的隔离。

docker有什么优势？
docker可以简化配置，将代码配置打包到镜像中随处可用
docker可以实现开发与生产环境的一致性，使用docker部署代码不再区分操作系统平台
docker可以更容易地将devops理念落地

docker的劣势？
隔离性差，正因为容器复用的宿主机的内核，且本质上还是应用程序之间的隔离，因此隔离性差。

说一下你对devops的理解？
我认为devops不试一种技术，而是一种运动，一种趋势，

如何使用docker临时启动一个容器？
docker run时加上--rm选项，容器关闭后就会删除

如何查看docker容器的日志？
docker容器的日志一般都会打印到标准输出中，使用docker logs 容器ID即可查看

Dockerfile中常用的命令？
FROM 后跟基础镜像
LABEL 可以给镜像打一些标签，比如作者名字
ENV 设置换进变量
COPY和ADD 可以将宿主机的文件拷贝到镜像中，COPY和ADD有些小区别
VOLUME 可以声明运行时将某一目录作为volume，实现数据持久化
WORKDIR可以指定工作目录
RUN 构建镜像时执行的命令
EXPOSE 声明要暴露的端口，正真暴露需要使用-p
ENTRYPOINT和CMD：容器启动时的命令，二者同时存在时CMD作为ENTRYPOINT的参数，ENTRYPOINT不可被替换

COPY和ADD的区别？
二者都可以将宿主机文件拷贝到镜像中。区别：ADD如果后跟tar.gz文件的话拷贝时会自动解压缩，且可以是远程文件，COPY可以在多阶段构建镜像时使用。一般我都用COPY

什么是docker镜像，什么是docker容器？
Docker镜像是Docker容器的源代码，用于启动docker容器。可以认为docker容器就是我们跑的应用，一般一个容器只跑一个应用

Docker有几种状态？
运行，退出，暂停，重新启动

你对devops的理解？
devops本身并不是一种技术，是一种运动，一种趋势 ，就是我们以往需要手工解决的问题，用工具化的方式来帮我们落地地解决问题，让我们突破dev和ops之间的屏障。
要实现Devops，不仅仅需要docker，k8s，还需要其他工具，比如版本管理工具，持续集成工具，监控工具。
开发人员需求确认-》代码修改-》代码提交-》测试-》测试通过-》代码merge到主分支-》触发持续集成-》自动构建镜像-》持续部署 实现代码在服务器上部署

Docker与虚拟机的区别？
简单理解为：虚拟机是操作系统级别的隔离，docker是应用级别的隔离，或者或进程间的隔离

Docker使用什么技术实现资源的隔离？
docker基于linux的namespace实现资源的隔离。linux有6个namespace：user ns，network ns，ipc ns，mounts ns，PID ns，UTS ns

Docker使用技术对资源控制？
cgroup

你对虚拟化技术的理解？
我认为虚拟化技术可以分为硬件层虚拟化和操作系统层虚拟化。比如kvm，vmware等技术都属于前者，而操作系统层虚拟化即容器技术，docker则是容器技术的典型代表

你是用docker部署过哪些应用？
之前帮公司部署的是nodejs应用和web网页，我自己部署过mattermost，sspanel，autopiano，人人影视等项目

npm install干了那些事？
npm install 会通过package.json文件寻找所需的函数库，也就是依赖，下载到本地，该命令执行完后可以看到本地多了一个node_module目录，安装的依赖文件都在这里面

docker-compose.yml文件中有哪些字段，说一下？
第一行写version，指定compose的版本，现在基本上都使用最新的3.x的版本
第二行写services，然后在里面定义我们要跑的应用，每个service就是一个容器，比如nginx，php，mysql等，然后每个service里面可以定义镜像名字或者Dockerfile文件位置，容器名字，要暴露的端口号，要持久化的目录，环境变量，失败后是否重启，是否依赖其他service，容器的网络等
如果容器要跑在swarm集群中的话，还可以使用deploy指定容器的副本数量，重启策略，容器可以跑在哪些节点上等
文件最后必须要声明我们使用的network，如果上面volume使用的定义的存储卷的话，下面也要声明

应用程序交付到运维手上之前，进行了什么过程？

Plan-> 架构->开发->构建->测试->(有问题打回去继续开发)->

持续集成过程：
开发人员开发完代码->提交到代码仓库->某工具自动触发，拖出来代码做构建->测试工具自动进行测试->没问题则进入下一步，否则报告给开发人员再次开发

持续交付：
如果测试完成后可以自动打包到可以被运维或者客户拿到的一个共享仓库中，让运维人员可以拿到打包好的，构建好的产品

持续部署：
运维拿到产品之后要做部署，加入交付完成后的产品，有一款工具可以自动将其拖出来，自动发布出去，这就叫持续部署

假如这一些列环节都可以自动实现，自动向前反馈：开发提交代码到仓库后，由工具触发自动进行构建，测试；出问题后自动反馈给开发修改，没问题则由工具触发自动交付；再由工具自动触发部署，部署出现bug再次反馈给开发，再次修复，发布。 这整个过程通常就叫做devops，打通了运维和开发的边界，甚至把运维都给干掉了。

以后这整个持续化的自动化的流水线的作业，就是运维工程师需要关注的，恰恰正是容器和编排工具的出现使得这一切终于落地了。
原因在于，构建好的产品基于容器构建的，不在区分客户或者说生产环境的操作系统平台差异，只要该平台能够运行docker容器，产品就能正确部署。这就是容器带来的好处。
